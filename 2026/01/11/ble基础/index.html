    BLE基础

    蓝牙广播

    概念

    ​ 低功耗蓝牙（BLE）一共有40个信道，其中只有3个用于信道用于蓝牙广播（37、38、39），频段范围聪2402MHz到24980MHz（2MHz一个信道）

    广播数据包（最大37字节）

    结构组成（即用户最大31字节可编辑）
    用户使用数据格式

    这31字节又被分为若干个广播数据结构体（AD_Structure），该结构体又有三个数据成员，分别是：

    • 长度（1个字节 其值 ＝ 1 + n（加上类型的1字节））
    • 数据类型（1个字节）
    • 数据（n个字节）（采用utf-8形式，所以可以显示中文，一个汉字3个字节）

    扫描响应（数据格式同用户自定义广播数据）

    概念
    • 可作为广播数据的补充（因为用户自定义广播数据最大31字节，但需收到扫描请求）
    • 非必需的数据且数据格式同用户自定义广播数据
    • 与广播数据包不同点：广播数据包是从机主动发送的，扫描响应需收到主机的扫描请求
    广播类型
    广播类型 可被连接 扫描响应 实际用途
    可连接非定向 √ √ 最常用
    可连接定向 指定设备 × 配对后快速连接
    不可连接非定向 × × 信标、传感器
    可扫描非定向 × √ 第三个的可扫描版本

    蓝牙状态切换

    • 可以看到我们需要再各种态切换的时候做出一些动作
    • 比如：在断开蓝牙后 设备进入就绪态，此时就搜寻不到蓝牙了。得继续让设备进入广播态才能被搜寻到

    服务、特征、描述符CCCD

    在蓝牙低功耗（BLE）开发中，服务（Service）、特征（Characteristic） 和 描述符（Descriptor） 是蓝牙设备数据交互的核心元素，它们通过 GATT 协议（Generic Attribute Profile）组织数据。以下是通俗易懂的解释：

    服务（Service）

    • 是什么：服务是蓝牙设备功能的逻辑分组，类似于一个”功能模块”。(eg：心率监测服务、电池电量服务)
    • 核心作用：定义设备能做什么（如测量心率、监控电量）。
    • 通过 UUID 唯一标识。

    特征（Characteristic）

    • 是什么：特征是服务中的具体数据点，用于实际的数据读写操作。(eg：心率值、电池电量百分比)
    • 核心作用：存储具体数据（如 71 bpm）。
    • 通过UUID 和 属性（Properties） 定义操作权限（读/写/通知等）。
    • 属性（Properties）示例：理解tips：以手机作为主体看待read，write（仅对于事件），notify则是设备端视角
      • READ：允许读取数据
      • WRITE：允许写入数据
      • NOTIFY：主动推送数据（如实时心率），单一发送不保证客户端收到
      • INDICATE：类似NOTIFY，但需客户端确认接收

    描述符（Descriptor）（CCCD）

    • 是什么：描述符是特征的附加信息，用于细化特征的行为或配置。（eg：启用数据通知、设置数据格式）

    • 核心作用：配置特征的行为（如开启实时通知）。补充描述特征（如单位、数据范围）。

    • 最常见描述符：
      Client Characteristic Configuration Descriptor (CCCD)
      用于启用/禁用特征的 NOTIFY 或 INDICATE 功能。

    三者的关系

    实际开发场景

    1. 读取电池电量
    • 找到电池服务（UUID 0x180F）
      → 读取特征 0x2A19（电池电量值）。
    1. 接收实时心率数据
    • 找到心率服务（UUID 0x180D）
      → 写入 CCCD 描述符（开启 NOTIFY）
      → 监听特征 0x2A37（心率值）。
    1. 控制智能灯泡
    • 创建自定义服务（128-bit UUID）
      → 添加特征（如 开关、颜色）
      → 设置特征属性为 WRITE。

    关键总结

    概念 类比 作用 典型 UUID 类型
    服务 手机的”相机”模块 功能分组 16-bit（标准服务）
    特征 相机中的”拍照按钮” 数据读写/操作入口 16-bit 或 128-bit
    描述符 按钮的”长按提示说明” 配置特征的附加信息 通常 16-bit

    配网应用层

    具体协议可看BLE配网协议.md

    基础知识铺垫

    蓝牙规范

    • 若一个 GATT 特征声明支持 Notify（通知）或 Indicate（指示），必须配套 CCCD 描述符（设备端必须建立好CCCD）；
    • Notify（通知）不需要应答，而Indicate（指示）需要应答
    • APP / 小程序作为「客户端」，必须先向设备的 CCCD 写入配置值（如 0x0001 开启 Notify、0x0000 关闭），设备端才能向 APP 主动推送 Notify 数据。

    初始化阶段（ble_prov_init 函数）

    基础初始化

    • 初始化 BLE 蓝牙核心模块，创建信号量用于同步 BLE 事件回调
    • 注册 GAP（蓝牙通用访问配置）和 GATTS（蓝牙通用属性服务端）回调函数，用于处理蓝牙连接、数据读写、MTU 更新等事件
    • 注册 GATTS 应用，等待注册完成事件

    设备信息配置

    • 获取蓝牙 MAC 地址，拼接成自定义设备名称（格式：LINKUP-XXYY，XXYY 为 MAC 后两位）

    • 设置 BLE 设备名称，对外广播该名称

      详细可看1.1蓝牙广播

    GATT 服务创建

    • 定义配网相关的 GATT 服务（UUID 0xFA00）和特征值：

      • 通知特征值（UUID 0xEA01）：用于向 APP 推送数据（WiFi 列表、配网结果）
      • 可写特征值（UUID 0xEA02）：用于接收 APP 下发的指令 / 配置

    • 创建 GATT 属性表，启动配网服务，等待服务启动完成事件

    广播配置与启动

    • 设置 BLE 广播参数（广播类型、间隔、信道、地址类型等）
    • 构造广播数据（包含 BLE 标志、配网服务 UUID、设备名称）
    • 启动 BLE 广播，对外宣告设备可连接

    连接与事件处理阶段（回调函数）

    连接管理
    • 设备被 APP 连接：记录连接 ID，初始化 MTU
      • MTU是蓝牙最大传输单元，如果双方都设置了，则采取双方最小值作为MTU
      • 如果其中一方或双方未设置，则采用默认MTU为23
      • MTU中有3个字节是固定ATT控制位，所以在默认下有效传输单元是20Bytes
    • 设备断开连接：重置连接状态，重新启动 BLE 广播
    • MTU 更新：接收 APP 的 MTU 协商请求，更新最大传输单元，优化数据传输效率
    数据读写处理
    • CCCD 写事件：APP 开启 / 关闭通知功能，记录通知使能状态
    • 可写特征值写事件：接收 APP 下发的协议帧，调用 ble_parse_proto_frame 解析

    协议帧解析与指令处理

    协议帧解析（ble_parse_proto_frame）
    • 校验协议帧头（0xB0）、长度、校验和，确保数据合法性
    • 处理分包：若为多包数据，拼接完整 JSON 数据；若为最后一包，触发指令处理
    APP 指令处理（ble_handle_app_message）
    • 指令类型 1（配网请求）：解析 JSON 中的 WiFi SSID、密码、加密类型，调用 ble_wifi_connect 连接 WiFi，反馈配网结果
    • 指令类型 2（扫描 WiFi 请求）：调用 ble_send_wifi_list_real 扫描周边 WiFi，将结果（SSID、RSSI、加密类型）推送给 APP
    • 指令类型 255（ACK 确认）：接收 APP 的确认消息，仅日志打印

    4. 数据推送阶段

    WiFi 扫描与结果推送
    • 注册 WiFi 扫描完成事件回调，启动 WiFi 扫描
    • 等待扫描完成，获取周边 WiFi 列表（最多 3 个），封装成 JSON 格式
    • 通过 BLE 通知特征值，分包推送 WiFi 列表给 APP
    配网结果推送（ble_send_prov_result）
    • 连接 WiFi 成功：推送设备 SN、版本、名称、MAC 等信息（类型 202）
    • 连接 WiFi 失败：推送失败标识（类型 201）
    • 所有推送数据按协议帧格式分包（支持 MTU 自适应），添加校验和，确保传输可靠

    总结

    核心步骤可简化为：

    1. 初始化：BLE 基础配置 → GATT 服务创建 → 广播启动；
    2. 交互：处理连接 / 断开 → 解析 APP 指令（WiFi 扫描 / 配网）；
    3. 反馈：执行 WiFi 操作 → 封装结果 → 分包推送至 APP。

    交互细节

    把整个 BLE 配网过程想象成：

    • 设备 = 一个「配网服务小店」（只做 WiFi 配网相关的事）
    • APP = 来店里办事的「顾客」
    • BLE 蓝牙 = 连接小店和顾客的「双向对讲机」
    • CCCD = 小店门口的「订阅开关」（顾客开了开关，小店才能主动喊顾客）

    设备与 APP 交互的完整流程（一步步拆解）

    第一步：设备「开店营业」（设备端初始化，对应代码ble_prov_init）

    设备启动后，会完成这些操作，准备好和 APP 通信：

    1. 搭好小店框架

      ：创建「配网服务柜台」（GATT 服务，UUID 0xFA00），柜台分 2 个窗口：

      • 「通知窗口」（UUID 0xEA01）：只出不进，小店（设备）主动喊顾客（APP），比如告诉顾客 “周边 WiFi 列表是这些”“配网成功 / 失败”；
      • 「写字窗口」（UUID 0xEA02）：只进不出，顾客（APP）写字条给小店，比如 “帮我扫一下周边 WiFi”“我要连 XX WiFi，密码是 XXX”。

    2. 装订阅开关：在「通知窗口」旁装一个开关（CCCD，UUID 0x2902），默认是关的 —— 顾客不打开，小店就算有消息也不能主动喊。

    3. 喊人来店：设备开启 BLE 广播，相当于小店在门口喊 “我是 LINKUP-XXYY，能配网，快来找我～”，广播里会告诉周边：我是谁（设备名）、我能办啥业务（配网服务 UUID）。

    第二步：APP「进店连接」（APP 主动连接设备）

    1. 发现小店：APP 打开蓝牙后，会扫描周边的 BLE 广播，看到设备喊的 “LINKUP-XXYY”，就知道这是能配网的设备；
    2. 建立连接：APP 选择这个设备，和设备建立 BLE 连接 —— 相当于顾客走进小店，和店员搭上了对讲机，能双向说话了；
    3. 协商 “说话效率”：APP 和设备会协商「MTU（最大传输单元）」，比如默认一次最多传 23 个字节，协商后可能传更多 —— 相当于约定好 “每次说话最多说多少字”，避免说太长得话断成太多截。

    第三步：APP「打开订阅开关」（操作 CCCD）

    APP 要接收设备主动推送的消息（比如 WiFi 列表），必须先开开关：

    1. APP 找到设备「通知窗口」旁的 CCCD 开关（对应代码里的s_cccd_handle）；

    2. APP 给开关写一个值：0x0001（开启通知）—— 相当于顾客告诉店员 “有消息直接喊我，不用我每次问”；

      • 如果写 0x0000，就是关闭通知，店员就算有消息也不能主动喊；
      • 设备端会记录这个开关状态（s_notify_enabled），只有开关开了，才会主动推送数据。

    第四步：双向「传消息」（核心交互，分 2 种场景）

    消息传递有个前提：设备和 APP 约定了「协议帧格式」，就像店员和顾客约定好 “写字条必须按固定格式写，不然看不懂”—— 协议帧格式是：头（0xB0 是 APP 发给设备，0xA0 是设备发给 APP）+ 类型 + 总包数 + 分包号 + 数据长度 + 实际数据 + 校验和。

    场景 1：APP 让设备扫 WiFi（APP→设备）

    1. APP 按协议帧格式，写一个 “扫 WiFi” 的指令（类型 2），通过「写字窗口」（0xEA02）发给设备；

      • 如果指令太长（比如超过 MTU 限制），会拆成多个包发；

    2. 设备收到数据后（

      函数）：

      • 先校验：是不是约定的帧头（0xB0）？长度对不对？校验和对不对？（防止消息传错）；
      • 分包拼接：如果是多个包，就拼成完整的 JSON 指令（比如 {“type”:2}）；

    3. 设备解析指令（ble_handle_app_message函数）：看到 type=2，就知道要扫 WiFi；

    4. 设备执行扫 WiFi 操作（

      函数）：

      • 启动 WiFi 扫描，等扫描完成后，整理周边 WiFi 列表（SSID、信号强度、是否加密）；
      • 把 WiFi 列表封装成 JSON，按协议帧格式拆包（ble_send_proto_text函数），通过「通知窗口」（0xEA01）推送给 APP；

    5. APP 收到设备推送的 WiFi 列表，展示给用户。

    场景 2：APP 让设备连 WiFi（APP→设备→反馈结果）

    1. APP 把用户输入的 WiFi 名称（SSID）、密码，按协议帧格式（类型 1），通过「写字窗口」发给设备；

    2. 设备解析指令（type=1），取出 SSID 和密码，尝试连接 WiFi（ble_wifi_connect函数）；

    3. 设备把连接结果（成功 / 失败）封装成 JSON：

      • 成功：推送给 APP “配网成功”+ 设备信息（SN、MAC 等，类型 202）；
      • 失败：推送给 APP “配网失败”（类型 201）；

    4. APP 收到结果后，告诉用户 “配网成功 / 失败”。

    第五步：结束交互（断开连接）

    1. 配网完成后，APP 可以主动断开 BLE 连接；或者设备长时间没交互，也会断开；
    2. 设备断开连接后，会重置状态（比如关闭通知开关、清空未处理的消息），然后重新开启广播，等待下一次连接。

    总结（小白版核心要点）

    1. 设备先 “广播喊人”，APP 主动连设备：BLE 是 “主从模式”，APP 是主，设备是从，必须 APP 先找设备；
    2. CCCD 是通知的 “总开关”：APP 不开这个开关，设备再想推送消息也发不出去；
    3. 消息要按 “约定格式” 传：不管是 APP 发还是设备发，都要包着头、类型、校验和等，防止传错、传丢；
    4. 长消息会拆包：一次传不完的话，会分成多个小包，接收方拼起来再解析。
